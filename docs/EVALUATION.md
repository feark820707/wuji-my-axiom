# 評估方法

> 量化框架的運作效果，用數據驅動校準。

## 四項核心指標

### 1. 推導準確率

AI 從太極 + 爻則推導出的行為，與使用者期望一致的比例。

```
推導準確率 = 正確推導次數 / 總推導次數
```

- **正確**：AI 的行為符合使用者期望，不需糾正
- **錯誤**：AI 的行為被使用者糾正
- **模糊**：AI 標記不確定交人判斷（不計入準確率，但追蹤頻率）

目標：穩定使用一週後 > 80%，一個月後 > 90%。

### 2. 校準收斂速度

從冷啟動到首次無糾正工作階段所需的天數。

```
收斂天數 = 首次零糾正工作日 - 冷啟動日
```

反映太極與爻則的初始品質。收斂越快，說明冷啟動校準越精確。

目標：7 天內收斂。超過 14 天考慮重新校準。

### 3. 爻則穩定度

一段時間內爻則的新增與刪除頻率。

```
爻則穩定度 = 1 - (爻則變動次數 / 觀察天數)
```

穩定度高表示系統已收斂。穩定度持續低表示太極不夠精確，爻則在反覆調整。

目標：使用一個月後，每週爻則變動 < 1 次。

### 4. 覆蓋率

新遇到的情境中，AI 能用太極 + 爻則自主處理（不需問人）的比例。

```
覆蓋率 = 自主處理次數 / (自主處理次數 + 交人判斷次數)
```

覆蓋率太低說明公理不夠通用。覆蓋率太高要警惕——可能是 AI 該問沒問。

目標：60-85%。低於 60% 需強化公理，高於 85% 需檢查是否有該交人沒交的。

## 日誌格式

每日記錄推導結果，格式如下：

```
## YYYY-MM-DD

| 時間 | 情境摘要 | 推導結果 | 類型 | 備註 |
|------|----------|----------|------|------|
| 09:15 | 收到模糊需求 | 先問 why | 正確 | |
| 10:30 | 刪除檔案前 | 直接刪除 | 錯誤 | 應先確認，不可逆操作 |
| 14:00 | 新情境無先例 | 交人判斷 | 模糊 | 太極可覆蓋但信心不足 |

統計：正確 8 / 錯誤 1 / 模糊 2 → 準確率 89%（不含模糊）
爻則變動：無
覆蓋率：8/10 = 80%
```

## 結果解讀

### 框架運作良好的訊號

- 推導準確率穩定 > 85%
- 一週內無爻則變動
- 覆蓋率穩定在 70-85%
- 錯誤集中在少數特定領域（可針對性加爻則）

### 需要調整的訊號

- 推導準確率 < 70%：太極可能不夠精確，考慮重新校準
- 爻則頻繁變動：爻則之間可能有衝突，需要壓縮
- 覆蓋率 < 50%：公理太抽象，推導不出具體行為
- 覆蓋率 > 90%：AI 可能過度自信，抽查是否有遺漏的交人時機
- 模糊比例 > 30%：AI 過度保守，可能需要更明確的爻則

### 調整方向

| 問題 | 調整 |
|------|------|
| 同類錯誤反覆出現 | 加一條針對性爻則 |
| 爻則太多 > 10 條 | 壓縮為更通用的原則 |
| 新領域覆蓋差 | 檢查太極是否足夠通用 |
| 校準遲遲不收斂 | 重跑冷啟動校準 |

## 評估頻率

- **每日**：記錄推導日誌（可讓 AI 自動記錄）
- **每週**：計算四項指標，識別趨勢
- **每月**：評估是否需要紀元轉換（見 [EVOLUTION.md](EVOLUTION.md)）

量化不是目的，校準才是。數據用來發現問題，發現後用變通機制解決。
