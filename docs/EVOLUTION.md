# 演化指南

> 系統不是建好就不動，而是持續演化。

## 紀元（Epoch）

紀元是系統的重大轉折點。不是版本號的遞增，而是治理邏輯的質變。

典型的紀元進程：

```
紀元 0：規則列舉期 — 用大量明確規則控制 AI 行為
紀元 1：壓縮過渡期 — 發現規則太多，開始合併歸納
紀元 2：公理生成期 — 從規則壓縮出公理，用推導取代列舉
紀元 N：持續精煉   — 公理與爻則的微調，覆蓋率持續提升
```

每個使用者的紀元進程不同，速度也不同。重點是方向：從列舉走向生成。

## 紀元轉換的訊號

以下徵兆表示當前紀元該結束了：

**規則膨脹**：爻則數量持續增加，超過 10 條仍在成長。規則在互相打架。

**原則衝突**：太極推導出的行為與爻則矛盾。根源可能是太極不夠精確，或爻則該被淘汰。

**覆蓋缺口**：頻繁遇到太極 + 爻則無法處理的情境，需要不斷問人。

**壓縮衝動**：你發現多條爻則其實在說同一件事，只是表述不同。

**價值轉變**：你的核心價值或生活重心發生根本變化，現有太極不再代表你。

## 演化方法

### 壓縮，不是增加

錯誤做法：遇到新情境 → 加一條規則。
正確做法：遇到新情境 → 問「現有公理為什麼沒覆蓋」→ 調整公理。

```
列舉 → 壓縮 → 生成
 多條規則 → 少數爻則 → 一條太極
```

規則數量應該隨時間減少，覆蓋率應該隨時間增加。如果規則在增加，方向錯了。

### 壓縮流程

1. 收集近期所有爻則與記憶中的回饋模式
2. 找出重複出現的深層原因
3. 嘗試用一句話覆蓋多條爻則
4. 驗證新公理是否能推導出被壓縮的爻則
5. 如果可以，刪除冗餘爻則，更新太極

### 紀元轉換步驟

1. **記錄現狀**：當前的太極、爻則、記憶快照
2. **分析不足**：哪些情境沒被覆蓋、哪些規則在衝突
3. **重新校準**：可選擇重跑冷啟動校準，或手動調整
4. **更新道層**：修改 CLAUDE.md 的太極與爻則
5. **清理德層**：被新太極覆蓋的記憶可以歸檔
6. **記錄轉換**：寫下紀元轉換的原因與變更內容

## 變通→演化的路徑

日常的變通機制是演化的燃料：

```
日常糾正/確認
     │
     ▼
 記憶累積（德層）
     │
     ▼  模式浮現
 爻則更新（道層微調）
     │
     ▼  爻則飽和
 太極壓縮（道層重構）
     │
     ▼  太極質變
 紀元轉換
```

不需要刻意推動演化。持續使用、持續回饋，系統自然走向壓縮。你要做的是在訊號出現時識別它，然後執行轉換。

## 紀錄紀元

每次紀元轉換，記錄以下內容：

```markdown
## 紀元 N → 紀元 N+1

日期：YYYY-MM-DD
觸發原因：{什麼訊號觸發了轉換}
主要變更：{太極/爻則的具體修改}
壓縮成果：{刪除了幾條規則，覆蓋率變化}
教訓：{這次轉換學到什麼}
```

紀元紀錄是框架最有價值的資產之一。它記錄了你的治理思維如何演化，對未來的調整有直接參考價值。
