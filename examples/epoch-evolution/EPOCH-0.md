# EPOCH-0：模仿期

> 虛構人物：林小明，28 歲，自由接案前端/後端工程師。目標是做出自己的 SaaS 產品，但被客戶案件卡住時間。

## 背景

林小明離開上一間新創公司兩年了，全職接案維生。他開始使用 AI 輔助工作，但沒有任何框架——每次開新對話就是一張白紙，AI 不知道他是誰、在做什麼、重視什麼。

他覺得「應該建一個系統」，但不知道從哪裡開始。

## 做了什麼

林小明直覺地搬來上一間公司的管理結構。那間公司有 8 個人，分前端組、後端組、設計組，有周會、有 code review 流程、有 Jira 看板。

他寫了一份 system prompt，大致長這樣：

```
你是我的技術團隊助理。我們的團隊結構如下：

【前端部門】
- 負責 UI/UX 開發
- 遵循 React 規範
- 每週五提交 code review

【後端部門】
- 負責 API 設計與資料庫
- 遵循 RESTful 規範
- 每日更新進度

【專案管理】
- 使用 Kanban 流程
- 每週一 sprint planning
- 每週五 retrospective

【品質標準】
- 測試覆蓋率 > 80%
- 文件必須與代碼同步
- PR 必須有至少一人 review
```

他還建了一個 Notion 看板，把自己拆成「前端角色」「後端角色」「PM 角色」，試圖讓 AI 分別扮演不同部門的人。

## 發生了什麼

### 第一週

AI 很配合地扮演「團隊助理」，但每次互動都充滿虛假的儀式感：

- AI 問：「需要我安排前端部門的 sprint 嗎？」（前端部門只有林小明自己）
- AI 說：「建議在下次 retrospective 討論這個技術債。」（retrospective 也只有他自己）
- AI 產出的文件充滿「團隊協作」語言，但閱讀者永遠只有一個人

### 第三週

Notion 看板已經荒廢。林小明發現自己花更多時間維護「管理系統」而不是寫程式碼。

一個客戶的後端 API 出了 bug，他需要快速修復。但他的 system prompt 說「PR 必須有至少一人 review」——review 的人只能是他自己透過 AI 模擬的。他覺得這很荒謬，但又不想打破自己定的規則。

### 第六週

他嘗試用 AI 來管理他的「個人 SaaS 產品」專案和「客戶案件」之間的時間分配。AI 按照「PM 角色」建議他做 sprint planning，排了整整兩小時。結果 plan 出來的排程在第二天就因為客戶臨時改需求而全部作廢。

### 第八週

他放棄了。把 Notion 看板關了，system prompt 清空，回到每次對話重新說明背景。

## 根本問題

**借來的結構不匹配現實。**

- 8 人團隊的管理結構套在 1 人組織上，產生的是摩擦而非效率
- 「部門」「review」「sprint」都是多人協作的抽象，一個人用是浪費
- 結構本身變成了負擔——維護結構的成本超過了結構帶來的價值
- AI 沒有被告知真正重要的事（他的價值觀、優先順序、決策方式），只被告知流程

## 數據

| 指標 | 數值 |
|------|------|
| 持續時間 | 約 2 個月 |
| system prompt 長度 | ~800 字 |
| 有效推導率 | 無法衡量（AI 在查表，不在推導）|
| 未覆蓋情境處理 | AI 亂猜或問使用者 |
| 維護成本 | 高（手動更新看板、流程文件）|

## 教訓

1. **結構必須從自身長出來，不能借。** 別人的組織結構解決的是別人的問題。
2. **一人組織不需要多人組織的管理開銷。** 沒有溝通成本的地方不需要溝通協議。
3. **AI 需要知道「為什麼」，不是「怎麼做」。** 流程是手段，價值觀才是根。
4. **系統的成本不能超過收益。** 如果維護系統比不用系統還累，系統就該廢。

---

*EPOCH-0 結束。林小明進入 EPOCH-1——從模仿別人的結構，轉向自己列舉規則。*
